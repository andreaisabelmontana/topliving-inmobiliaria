01 · Ciclo de vida

SDLC: modelo iterativo-incremental

Top Living se desarrolla en iteraciones cortas (≈2 semanas). Cada iteración entrega valor en producción: primero el catálogo estático, luego búsqueda, luego detalle, luego formularios. El modelo iterativo encaja con un equipo pequeño que necesita feedback temprano del cliente y reducir riesgo de integración tardío.

Justificación: Evitamos Waterfall porque los requisitos cambian (nuevos tipos de inmueble, integraciones con MLS). Evitamos Big-Bang porque queremos poder devolver versiones a producción cada semana sin reentrenamiento.
02 · Control de versiones

Git: trunk-based con PRs cortos

Ramas vivas: main (siempre desplegable), ramas feat/*, fix/*, chore/* que viven menos de 48h y se mergean vía PR con revisión obligatoria.

$ git checkout -b feat/property-filters
$ git add js/app.js properties.html
$ git commit -m "feat(listings): add city + sort filters"
$ git push -u origin feat/property-filters
$ gh pr create --base main --fill

Convenciones de commit: feat, fix, refactor, test, docs, chore (Conventional Commits).

03 · Arquitectura

N-tier con frontera hexagonal para el dominio

Tres capas claras + un núcleo de dominio aislado:

Capa 1Frontend estático
HTML · CSS · JS
Capa 2API Flask
REST + JSON
NúcleoDominio puro
Property, Lead
Capa 3Persistencia
SQLite / Postgres

El dominio no importa de Flask ni de la BD — recibe puertos (interfaces). Esto permite cambiar SQLite por Postgres sin tocar reglas de negocio y testear el dominio sin servidor ni base de datos.

Monolito modular → futuro microservicios

Hoy es un monolito modular (un solo proceso con módulos catalog, leads, monitoring). Si el módulo leads crece, se extrae como microservicio detrás del mismo gateway sin cambiar el frontend.

05 · Calidad de código

SOLID + bad-practice budget = 0

  • Single Responsibility: handlers HTTP no formatean precios.
  • Open/Closed: añadir un nuevo tipo de inmueble = añadir un valor al enum, no editar 5 archivos.
  • Dependency Inversion: el servicio recibe el repositorio por constructor.
  • No magic numbers: umbrales en settings.py.
  • No duplicación: formato de precio en una sola función (formatPrice).
Linters configurados: ruff + black (Python), eslint + prettier (JS). El pipeline falla si el linter falla.
06 · Testing

Pirámide de tests · cobertura ≥ 70%

UnitLógica de dominio
pytest · ~85% del total
IntegraciónAPI + DB
pytest + Flask test client
E2EFlujo crítico
Playwright (smoke)

El pipeline mide cobertura con pytest --cov y rompe el build si la cobertura cae por debajo del 70%. Reporte HTML publicado como artifact y como comentario en cada PR.

# tests/test_property.py
def test_price_per_m2_is_computed():
    p = Property(price=980_000_000, area=112)
    assert round(p.price_per_m2) == 8_750_000

def test_filter_by_city_is_case_insensitive(repo):
    repo.add(Property(city="Bogotá"))
    assert len(repo.search(city="bogotá")) == 1
07 · CI/CD

Pipeline en GitHub Actions

1Checkout
2Lint
3Tests + cov
4Build Docker
5Security scan
6Deploy
# .github/workflows/ci.yml — resumen
name: ci
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r requirements.txt
      - run: ruff check .
      - run: pytest --cov=app --cov-fail-under=70
      - uses: docker/build-push-action@v5
        with:
          push: ${{ github.ref == 'refs/heads/main' }}
          tags: ghcr.io/${{ github.repository }}:latest
  deploy:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - run: flyctl deploy --remote-only

Sólo main dispara despliegue. Secretos en GitHub Secrets (FLY_API_TOKEN, DATABASE_URL).

08 · Contenedores

Docker multi-stage, imagen < 90 MB

# Dockerfile (extracto)
FROM python:3.12-slim AS build
WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

FROM python:3.12-slim
WORKDIR /app
COPY --from=build /root/.local /root/.local
COPY app/ ./app
ENV PATH=/root/.local/bin:$PATH
USER 1001
EXPOSE 8000
HEALTHCHECK --interval=30s CMD curl -f http://localhost:8000/health || exit 1
CMD ["gunicorn", "-b", "0.0.0.0:8000", "app.main:app"]

Imagen distroless-friendly, usuario no-root, healthcheck nativo de Docker.

10 · Seguridad

Defensa por capas

  • Secrets: nunca en el repo. GitHub Secrets + .env en local (gitignored).
  • Dependencias: pip-audit en CI; Dependabot abre PRs automáticos.
  • Imagen Docker: escaneo con trivy antes del push.
  • Transporte: HTTPS forzado, HSTS, certificados gestionados por Fly.io / Cloudflare.
  • OWASP top 10: queries parametrizadas, validación de inputs con Pydantic, rate-limit en /leads.
  • Roles: contenedor corre como usuario no-root (USER 1001).
11 · Proceso

Scrum simplificado

Sprints de 2 semanas. Backlog en GitHub Projects. Ceremonias: planning (lunes), daily async (Slack), review + retro (viernes). DoD: tests verdes, cobertura ≥ 70%, revisado por al menos 1 peer, deployado a staging.
